My first OpenXava project
If you are a newcomer to OpenXava, look at the quick start guide first.
As alternative to this tutorial you can watch the Creating a new OpenXava project in Eclipse video.

Create a new project

First open your Eclipse IDE and set the workspace to the one that comes with the OpenXava distribution. Then:
  • Using the appropriate Eclipse Wizard, create a new empty Java Project in the workspace named Management.
  • Go to the project OpenXavaTemplate and execute CreateNewProject.xmlusing Ant:
    • You can do it with Right Button on CreateNewProject.xml > Run as > Ant Build.
    • Ant will ask you for the project name, enter "Management".
  • Select the project Management and refresh by pressing F5.

Now you have a new project ready to start working, but before continuing you need to configure the database.

Configure database

OpenXava generates a JavaEE/J2EE application intended to be deployed in a Java Application Server (since v2.0 OpenXava applications also runs in a simple servlet container, such as Tomcat). You only need to indicate the Data Source JNDI and then configure the data source in your application server. Configuring a data source in an application server is out of the scope of this guide, nevertheless you have detailed instructions below to configure a database in order to run this first project using the Tomcat included in OpenXava distribution as application server and Hypersonic as database. This Tomcat is in the folder openxava-5.x/tomcat.
With Tomcat stopped edit the file context.xml in Tomcat's directory conf. In this file add the following entry:
<Resource name="jdbc/ManagementDS" auth="Container" type="javax.sql.DataSource"
    maxActive="20" maxIdle="5" maxWait="10000"
    username="sa" password="" driverClassName="org.hsqldb.jdbcDriver"
    url="jdbc:hsqldb:hsql://localhost:1666"/>
 
The main thing here is the JNDI name, in this case ManagementDS. This is the only thing referenced from OpenXava. The driverClassName and url attributes depend on your particular database, Hypersonic in this case.

Your first business component

Creating an OpenXava business component is easy: the definition of each component is a Java class with annotations. In order to begin you have to create a class called Warehouse:
  • Put the mouse on src folder and use the Right Button > New > Package
  • Create a package named org.openxava.management.model
  • Put the cursor on package org.openxava.management.model and use Right Button > New > Class
  • Create a class named Warehouse
Now edit your new class and write this code:
package org.openxava.management.model;
 
import javax.persistence.*;
import org.openxava.annotations.*;
 
@Entity
public class Warehouse {
 
    @Id @Column(length=3) @Required
    private int number;
 
    @Column(length=40) @Required
    private String name;
 
    public int getNumber() {
        return number;
    }
 
    public void setNumber(int number) {
        this.number = number;
    }
 
    public String getName() {
        return name;
    }
 
    public void setName(String name) {
        this.name = name;
    }
 
}
This class contains (and it will always contain) all the details needed by the application about the Warehouse business component. At the moment we only have the data structure, but in this class you can put the mapping against the database, the business logic, the visual presentation, the tabular data management, etc.

This class is actually an Entity that follows the Java Persistence API (JPA) standard. For defining a class as an entity you only need to use the @Entity annotation in the class declaration, as follows:
@Id                                  // 1
@Column(length=3)                    // 2
@Required                            // 3
private int number;                  // 4
private int getNumber() {            // 4
    return number;
}
private void setNumber(int number) { // 4
    this.number = number;
}
This is the meaning of the annotations:
  1. @Id: Indicates if this property is part of the key. The key is a unique identifier of the object and usually matches with the primary key of a database table.
  2. @Column(length= ): Length of data. It's optional, but it's useful for displaying better user interfaces and generating database tables.
  3. @Required: Indicates if it's required to validate the existence of data for this property just before creation or modification.
  4. The property defined in the usual way for a Java class. All valid types for a Java property are applicable here, including integrated types, JDK classes and custom classes.

The possibilities of the property definition go far from what is shown here. You can see a more complete explanation in the Model chapter.

The table

Before testing the application you have to create the table in the database. Follow these steps:
  • Start your database server: From command line go to openxava-5.x/tomcat/bin folder and execute:
    • In Linux/Unix: ./start-hsqldb.sh management-db 1666
    • In Windows: start-hsqldb management-db 1666
  • Create the table:
    • Edit Management/build.xml. Search the ant target updateSchema.
    • Enter the correct value for schema.path, in this case “../OpenXavaTest/lib/hsqldb.jar”. (Not needed since v5.3.1)
    • Press Ctrl-B to build the project.
    • Execute the ant target updateSchema with right button on Management/build.xml and choose Run As > Ant Build...
  • Start Tomcat (tomcat/bin/startup.sh or .bat). If it is already started, stop (tomcat/bin/shutdown.sh or .bat) and restart it. Now everything is ready.

Executing your application

After all your hard work now it's time to see the fruit of your sweat. Let's go.
  • Press Ctrl-B to build the project.
  • Execute the ant target deployWar with right button on Management/build.xml and choose Run As > Ant Build...
  • Open an internet browser and http://localhost:8080/Management. It will ask you for user and password, enter admin/admin.
    • With v4.x use http://localhost:8080/Management/modules/Warehouse
    • With v3.x (really until v4m5) use http://localhost:8080/Management/xava/module.jsp?application=Management&module=Warehouse
And now you can play with your module and see its behavior.

Automating the tests

Although it seems that the most natural way to test an application is to open a browser and use it like an end-user; in fact it is more productive to automate the tests. As your system grows, you have it tied and you avoid breaking it when you advance.
OpenXava uses a test system based on JUnit and HtmlUnit. The OpenXava JUnit tests simulate the behavior of a real user with a browser. This way you can replicate exactly the same tests that you can do directly with an internet browser. The advantage of this approach is that you can easily test all layers of your program, from user interface to database.
When you test a module manually you usually create a new record, search it, modify and finally delete it. Let's do this automatically.
First you have to create a package for the test classes, org.openxava.management.tests, and then add the WarehouseTest class to it, with the following code:
package org.openxava.management.tests;
 
import org.openxava.tests.*;
 
/**
 * @author Javier Paniza
 */
 
public class WarehouseTest extends ModuleTestBase {
 
    public WarehouseTest(String testName) {
        super(testName, "Management", "Warehouse");  // 1
    }
 
    public void testCreateReadUpdateDelete() throws Exception {
        login("admin", "admin");                     // 2  New in v5.0
        // Create
        execute("CRUD.new");                         // 3
        setValue("number", "7");                     // 4
        setValue("name", "JUNIT Warehouse");
        execute("CRUD.save");
        assertNoErrors();                            // 5
        assertValue("number", "");                   // 6
        assertValue("name", "");
 
        // Read
        execute("CRUD.search");
        setValue("number", "7");
        execute("Search.search");
        assertValue("number", "7");
        assertValue("name", "JUNIT Warehouse");
 
        // Update
        setValue("name", "JUNIT Warehouse MODIFIED");
        execute("CRUD.save");
        assertNoErrors();
        assertValue("number", "");
        assertValue("name", "");
 
        // Verify if modified
        execute("CRUD.search");
        setValue("number", "7");
        execute("Search.search");
        assertValue("number", "7");
        assertValue("name", "JUNIT Warehouse MODIFIED");
 
        // Delete
        execute("CRUD.delete");
        assertMessage("Warehouse deleted successfully"); // 7
    }
 
}
To run this test click the mouse right button on WarehouseTest and choose Run As > JUnit Test.
You can learn from this example:
  1. Constructor: In the constructor you indicate the application and module name.
  2. login: (New in v5.0) Login in the application using a user and password.
  3. execute: Simulates a button or link click. You send the action name as an argument. You can view the action names in OpenXava/xava/default-controllers.xml (the predefined controllers) and Management/xava/controllers.xml (the customized ones). Also if you move the mouse over the link your browser will show you the JavaScript code with the OpenXava action to execute. For example, execute(“CRUD.new”) is like clicking on the 'new' button in the user interface.
  4. setValue: Assigns a value to a form control. That is, setValue(“name”, “Pepe”) has the same effect as typing the text “Pepe” in the field 'name'. The values are always alphanumeric because they are assigned to a HTML form.
  5. assertNoErrors: Verifies that there are no errors. In the user interface errors are red messages showed to user and added by the application logic.
  6. assertValue: Verifies if the value in the form field is the expected one.
  7. assertMessage: Verifies if the application has shown the indicated informative message.
You can see that it is very easy to test module's behavior; writing this code can take 5 minutes, but at the end you will save hours of work, because from now on you can test your module just in 1 second. If you break the Warehouse module (maybe touching another part of your application) your test warns you just in time.
For more details have a look at the JavaDoc API of org.openxava.tests.ModuleTestBase and examine the examples in org.openxava.test.tests of OpenXavaTest.
You can disable login in your application, in that way you can avoid the first line of the test, the login("admin", "admin") (2) and get better performance for tests. To do it edit the properties/naviox.properties file and write:
autologinUser=admin
autologinPassword=admin
This is also useful for development or public applications.

The labels

Now everything works well, but a little detail still remains. Maybe you want to define the labels to be shown to the user. The way is to write a file with all the labels so you can translate your product to another language with no problems.
To define the labels you only have to edit the file Management-labels_en.properties in i18n folder and add:
Warehouse=Warehouse
You do not have to put all the properties, because the more common cases (number, name, description and a big etc) are already included in English, Spanish, Chinese, Russian, German, Polish, Indonesian, Serbian, Swedish, French, Italian and Catalan.
If you wish the version in an other language (Spanish for example), you only need to copy and paste with the appropriate suffix. For example, you can have a Management-labels_es.properties with the following content:
Warehouse=Almacén
The default labels and messages are in OpenXava/i18n/Labels.properties and OpenXava/i18n/Messages.properties. If you want to override some of these resources you do not need to edit these files, instead, you can use the same key names in the resource files of your project, then your labels and messages will be used instead of the standard ones of OpenXava (new in v2.0.3). For example, if you want to replace the standard message in the list mode ''There are 23663 objects in list” with another, you add this entry to Management-messages_en.properties:
# list_count is in Messages_en.properties of OpenXava, this is an example
# of overriding a standard openxava message
list_count=There are {0} records in list
Now, your application will show “There are 23663 records in list” instead of the default OpenXava message “'There are 23663 objects in list”.
If you want to know more about how to define labels of your OpenXava elements please look in OpenXavaTest/i18n.
